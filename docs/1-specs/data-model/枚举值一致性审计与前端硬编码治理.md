## 一、问题总览

| 问题类别 | 数量 | 严重程度 |
|---|---|---|
| Entity 字段缺失字典类型 | 19 个字段无 ins_dict 对应条目 | **P1 中** |
| 后端代码与 DB 字典语义不一致 | 2 处（payment_status=4/5, claim_status=3/6） | **P0 高** |
| 前端硬编码与 DB 字典严重错位 | 2 处（Dashboard + ClaimList claim_status） | **P0 高** |
| 前端硬编码编码偏移 | 1 处（ProductList.vue versionStatus） | **P0 高** |
| 前端硬编码未使用 dictStore | 10 处 Record/数组 + 8 处 el-option | **P1 中** |
| 前端遗留状态下拉框 | 2 处（ChannelList/Detail 状态变更弹窗） | **P1 中** |

**审计方法**：
1. 扫描全部 Entity Integer 字段（80+ 个）→ 对比 ins_dict 44 组字典类型
2. 扫描后端 Service 代码中所有 setStatus()/setXxx() 硬编码值 → 对比 DB 字典
3. 扫描前端 Record<number, {label}> + statusOptions + el-option 硬编码

**根因**：
- 后端：部分状态码语义在代码注释和 DB 字典中定义不同，部分业务枚举从未入库
- 前端：多处使用硬编码映射表而非 dictStore，字典变更后前端不同步

## 六、治理原则（强制执行，禁止跳过）

### 原则 1：前端禁止硬编码映射表

前端状态码/类型码的 label 和 tag type **必须**通过 `dictStore.getDictLabel()` 获取。

- 禁止在前端文件中硬编码 `Record<number, {label, type}>` 映射表
- 禁止在前端文件中硬编码 `statusOptions` 数组
- 禁止在前端 `<el-option>` 中硬编码状态值选项
- 新增枚举值时只需更新 `ins_dict` 表 + Redis 缓存，前端自动生效

### 原则 2：后端优先原则（字典不一致时的修复决策规则）

当后端代码硬编码的状态值与 `ins_dict` 表不一致时，**必须先确认后端代码的实际运行行为**，再决定修复方向：

```
审计发现不一致时：
  Step 1：确认后端的代码值在生产中代表什么含义（运行时行为 = 真实语义）
  Step 2：判断后端行为是否符合业务预期
    - 符合 → 以代码为准，更新 ins_dict 表对齐（成本最低：只改 DB 几条记录）
    - 不符合 → 这是 Bug，以 ins_dict 为准，修正后端代码
  Step 3：无论哪种情况，前端一律改用 dictStore 获取 label
```

**禁止行为**：
- 禁止不审计直接假设"DB 是对的"或"代码是对的"
- 禁止跳过 Step 1 直接修改
- 禁止前端自行定义与后端/DB 不同的状态码映射

### 原则 3：三方数据一致性（DB → Redis → 前端）

```
ins_dict 表 (MySQL, 唯一权威源)
    → Redis 缓存 (运行时高速查询，DB 变更时自动刷新)
        → 前端 dictStore (API 拉取，存入 Pinia，页面渲染使用)
```

- `ins_dict` 是唯一权威源，Redis 和前端均为其派生数据
- 后端修改 `ins_dict` 后必须同步清除对应 Redis 缓存 key
- 前端禁止缓存状态码 label 到本地变量，每次渲染均从 dictStore 获取
