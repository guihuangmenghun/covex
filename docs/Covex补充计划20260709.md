## P1 — 权限编码规范统一（edit vs create/update）

### Task 1.0 权限编码模型统一：`update` → `edit` [Refactor]

- **问题**：数据库权限编码使用 `update`，与路由权限矩阵 W（编辑=创建+修改合一）语义不一致，且 Controller 注解已统一为 `edit`
- **根因分析**：
  1. **路由权限矩阵**（`Covex路由权限矩阵.md` v3.0）定义了 W = 可创建/编辑，即 create + edit 为一个整体权限
  2. **数据库种子数据**（V4）使用了 `update` 作为动作编码（如 `product:update`），与矩阵 W 语义不匹配
  3. **Controller 注解**已统一为 `edit`（如 `product:edit`），与数据库 `product:update` 不一致导致权限校验失败
  4. **后期补充权限**（V4 id=31）又用了 `user:edit`，与已有的 `user:update` 形成重复
- **为什么拆分是错的**：
  - 查看 V20 角色权限预设，所有角色的 `xxx:create` 和 `xxx:update` 都是**同时授予、同时撤销**，从未单独使用
  - 路由权限矩阵的 W 本身就是“创建+编辑”一体化语义，拆分后反而产生冗余
  - 拆分导致权限总数膨胀（63 vs 40），增加管理复杂度
- **建议方案**：统一使用 `edit` 作为 W 的权限动作编码，中文名为“编辑”
  - `product:update` → `product:edit`
  - `role:update` → `role:edit`
  - `channel:update` → `channel:edit`
  - `claim:update` → `claim:edit`
  - `proposal:update` → `proposal:edit`
  - `dict:update` → `dict:edit`
  - `policy:update` → `policy:edit`（待新增）
  - `user:update` → `user:edit`（已完成，id=3）
  - 删除重复的 id=31（已完成）
- **影响范围**：
  - 数据库：`ins_permission` 表所有 `*:update` 改为 `*:edit`
  - V20 角色权限预设 SQL 重新生成
  - 前端：`v-permission` 指令中的权限编码
- **预估**：1 天
- **验证**：
- [ ] 全项目扫描无 `*:update` 残留（`edit` 统一替代）
- [ ] Controller 权限注解与 `ins_permission` 表编码一致
- [ ] V20 角色权限预设重新生成并验证
- [ ] 前端权限指令编码同步更新
