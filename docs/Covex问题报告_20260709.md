# Covex 问题报告：StackOverflowError + 支付→出单链路断裂

> 日期：2026-07-09 | 影响范围：全局 API + 投保→出单 E2E 流程

---

## 一、问题 1：DataPermissionInterceptor StackOverflowError

### 1.1 错误位置

```
POST /api/user/login、POST /api/proposal 等所有需要登录的接口

调用链：
  业务查询 → MybatisPlusInterceptor.intercept()
    → DataPermissionInterceptor.beforeQuery()
      → PermissionCacheService.getDataScope()
        → UserService.getUserRoles()
          → userRoleMapper.selectList()
            → MybatisPlusInterceptor.intercept() ← 递归回第一步
```

**错误日志**：`covex-web/logs/error.log`，搜索 `StackOverflowError`

### 1.2 根因分析

| 阶段 | 代码状态 | 是否递归 |
|------|---------|---------|
| 初始版本 (`bf2245b`) | `beforeQuery()` 无 SKIP 防重入 | **是** |
| 修复版本 (`c5a2d23`) | 顶部加 `SKIP.get()` 检查 | 否 |
| 服务器实际运行 | 跑的是初始版本（未 `mvn install`） | **是** |

**核心原因**：代码修复已提交到 git（commit `c5a2d23`），但执行 `mvn spring-boot:run -pl covex-web` 时，covex-web 从本地 Maven 仓库加载 covex-service 的旧 JAR，而不是 target/ 目录的最新编译产物。

### 1.3 解决方案

```bash
# 修改非启动模块后，必须先 install 再启动
mvn install -pl covex-common,covex-api,covex-service -DskipTests
mvn spring-boot:run -pl covex-web
```

### 1.4 修复过程踩坑

| 尝试 | 方案 | 结果 | 原因 |
|------|------|------|------|
| 第 1 次 | InterceptorIgnoreHelper | ❌ 失败 | 未 `mvn install`，服务器跑的还是旧代码，误判方案不行 |
| 第 2 次 | LambdaQueryWrapper → QueryWrapper | ❌ 无效 | 同上，改的代码根本没部署上去 |
| 第 3 次 | `mvn install` + 重启 | ✅ 通过 | SKIP 机制本身是正确的，只是服务器没拿到新代码 |

**教训**：排查问题时先确认"代码是否真正部署了"，再判断方案是否正确。

## 二、问题 2：支付→出单链路断裂

### 2.1 错误位置

```
PaymentService.doHandlePaymentCallback()
  ├── ins_payment.status = 2（已支付）✅
  ├── ins_proposal.status = 5（已支付）✅
  └── 触发 issuePolicy() 出单 ← ❌ 缺失，代码到此结束
```

### 2.2 根因分析

**开发计划 S10 的任务拆解**：
```
- [x] 实现支付回调处理（幂等）      ← PaymentService ✅ 已实现
- [x] 实现出单链（LiteFlow issue）  ← PolicyService ✅ 已实现
- [x] 实现异步通知（RocketMQ）      ← IssueNotifyComponent ✅ 已实现
```

三个独立任务都完成了，但**没有"集成任务"把它们串起来**。

**需求规格写了**："已支付 → 触发 LiteFlow issue 链"，但开发计划没有把这个连接动作拆成具体任务。

### 2.3 解决方案

采用异步 MQ 方案（需求规格 Story 3.3 AC-2/AC-3/AC-4）：

```
PaymentService.handlePaymentCallback()
  ├── proposal.setStatus(5)
  └── rocketMQTemplate.convertAndSend("PROPOSAL_PAID", proposalId=xxx)
        ↓
ProposalPaidConsumer.onMessage()
  └── policyService.issuePolicy(proposalId)
        ├── proposal.setStatus(6)
        ├── 创建保单 ins_policy
        └── 发送 POLICY_ISSUED MQ → 佣金记录
```

### 2.4 修复过程踩坑

| 阶段 | 发生了什么 |
|------|-----------|
| 发现 | E2E 测试投保单卡在 status=5，保单未生成 |
| 误判 1 | 以为 PaymentService 漏了一行同步调用 → 用户纠正：应该看需求分析 |
| 误判 2 | 以为前端缺"出单"按钮 → 查代码发现前端 API 函数已定义但无调用 |
| 定位 | 全量搜索 `issuePolicy` 调用方 → 只有 Controller 暴露，无任何调用者 |
| 确认 | 需求规格明确要求自动触发，开发计划漏了集成任务 |
| 修复 | 更新需求规格 → 补充计划加 Task 3.5 → 实现 MQ 方案 → 字典表同步 |

**教训**：改代码前先查需求规格，确认"应该怎么连"而不是"哪里漏了补哪里"。

## 三、过程问题总结

### 3.1 多 Agent 协作冲突

- 两个 AI Agent 同时操作同一代码库，互相覆盖修改
- Agent A 注释了 `@Aspect`（调试），Agent B 不知道，导致权限校验被意外关闭
- **建议**：Agent 之间通过 git 分支隔离，或明确文件分工

### 3.2 诊断方法论错误

- 过早下结论（以为是 OGNL/lambda 问题），没有先确认"代码是否部署成功"
- 花大量时间分析框架源码，而不是先验证基础前提
- **建议**：遵循"先确认部署 → 再看日志 → 再查代码"的排查顺序

### 3.3 文档与代码脱节

- 需求规格写了"支付→触发 issue 链"，但开发计划没有对应任务
- 开发计划标了 [x] 的三个子任务都做了，但没人检查它们是否连起来
- **建议**：每个 AC 必须逐条映射到开发计划任务，尤其是跨模块集成点

## 四、新增/强化的项目规范

### 规范 1：多模块构建规范（已写入 AGENTS.md + 记忆）

```
修改 covex-service/covex-common/covex-api 后：
  ① mvn install -pl covex-common,covex-api,covex-service -DskipTests
  ② 再 mvn spring-boot:run -pl covex-web
禁止只 compile 不 install 就启动服务。
```

### 规范 2：需求 AC → 开发计划映射（已写入记忆）

```
需求规格中每新增/修改 AC，必须：
  ① 在开发计划或补充计划中找到对应任务
  ② 如无对应任务，必须新建任务条目
  ③ 重点关注跨模块/跨服务的集成点（A→B 的连接谁来写）
```

### 规范 3：字典表同步规范（已写入 AGENTS.md）

```
新增/修改状态码、类型码等枚举值时：
  ① 修改 V2__dict_seed_data.sql（新部署生效）
  ② 创建 V{N}__xxx.sql 增量迁移（已有数据库生效）
  ③ 前端显示文本以字典表为准，禁止硬编码
```

### 规范 4：问题排查三步法（建议写入 AGENTS.md）

```
1. 确认部署：改的代码是否真正部署到了运行环境？
   - 检查 mvn install 是否执行
   - 检查日志中的代码行号是否与源码一致
2. 查日志：错误堆栈指向哪里？哪个类的哪一行？
3. 查需求：需求规格对这块的描述是什么？应该怎么做？

禁止：跳过第 1 步直接分析源码。
```

### 规范 5：跨服务集成检查清单（建议写入 AGENTS.md）

```
开发计划中涉及"A 触发 B"的任务时，必须检查：
  □ A 的代码中是否有触发 B 的调用/MQ/事件？
  □ B 的消费者/监听器是否已创建？
  □ 失败时 B 是否有兜底策略？
  □ 端到端测试是否覆盖了 A→B 的完整链路？
```
